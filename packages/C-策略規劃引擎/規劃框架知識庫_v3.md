# 規劃框架知識庫 v3

> 本檔提供「策略規劃引擎」可用的規劃方法論。AI 會依使用者的時間長度與目標性質，自動選擇最合適的框架。
>
> 未來想替換或新增規劃框架（例如改用 OGSM、Hoshin Kanri、Balanced Scorecard、Kanban），直接修改本檔即可。

---

## 一、為什麼是 SMART + OKR 雙框架

策略規劃引擎刻意只用兩個框架，理由是：

### 1.1 完整覆蓋規劃需求

| 時間長度 | 框架 | 拆解粒度 |
|---|---|---|
| 1-2 週短衝 | SMART | 拆到「日」 |
| 1-3 個月 | OKR | 拆到「週」（雙週 sprint） |
| 3-6 個月 | OKR | 拆到「週」（月度檢視） |
| 6 個月以上 | OKR | 拆到「月」（月+季末檢視） |

OKR 一個框架就涵蓋從 1 個月到 1 年以上的所有中長期規劃。**框架不變、PDCA 節奏依時長變**。

### 1.2 OKR 的「補強做法」（四件套）

OKR 原本只有 O + KR 兩元素，如果直接拿來用，跟 PDCA 會有 gap：

| 問題 | 為什麼是問題 |
|---|---|
| 缺乏「執行路徑」層 | KR 設好後不知道怎麼做 |
| 缺乏「過程訊號」層 | 只能檢核「達成或沒達成」，不知道過程順不順 |

**解法**：擴充 OKR 為「四件套」（這是 OKR 實務界的標準擴充，例如 John Doerr 的《OKR：做最重要的事》中也提到 Initiatives）。

| 元素 | 對應 PDCA 的角色 |
|---|---|
| O（目標） | P（計畫）的方向 |
| KR（關鍵成果） | C（檢核）的結果指標 |
| **Action Items（行動方案）** | **D（執行）的具體做法** |
| **Leading Indicators（進度訊號）** | **C（檢核）的過程指標** |

## 二、框架選擇邏輯

| 時間長度 | 使用框架 | 核心節奏 | 適合特徵 |
|---|---|---|---|
| **1 至 2 週短衝** | **SMART × 日 PDCA** | 每日推進 | 明確產出物、緊迫、單一目標 |
| **1 至 3 個月** | **OKR × 雙週 PDCA** | 雙週迭代 | 多階段、需要協調、有方向但要拆分 |
| **3 至 6 個月** | **OKR × 月 PDCA** | 月度檢視 | 持續累積、需要中期調整 |
| **6 個月以上** | **OKR × 月 PDCA + 季末大檢視** | 月度 + 季末 | 長期累積、需要大方向校準 |

### 流程圖

```
        使用者描述目標 + 時間
              ↓
       【判斷時間長度】
       ↓             ↓
     1-2 週        1 個月以上
       ↓             ↓
     SMART         OKR
       ↓             ↓
                 【判斷時長細節】
                 1-3 月 / 3-6 月 / 6 月以上
                 ↓     ↓        ↓
                雙週   月      月+季末
                PDCA   PDCA    大檢視
```

### 判斷關鍵

- 時間 ≦ 2 週 → SMART
- 時間 1 個月以上 → OKR（再依時長定 PDCA 節奏）
- 使用者沒提時間 → 主動詢問選項
- 模糊描述（「快點做」「持續推進」） → 主動詢問

## 三、框架 1：SMART × 日 PDCA 短衝循環

### 適用情境

1 至 2 週內要完成的明確產出。例如：
- 一週內完成跨部門報告
- 兩週內推動 feature 上線
- 10 天內修復重大 bug 並驗證
- 一週內寫完論文章節
- 兩週內準備一場簡報

### SMART 目標設定（五要素）

| 字母 | 元素 | 引導問題 |
|---|---|---|
| **S** | Specific 具體 | 完成什麼具體產出物？解決什麼具體問題？ |
| **M** | Measurable 可衡量 | 完成時你會看到什麼數據／狀態？ |
| **A** | Achievable 可達成 | 以現有資源，這個目標真的能在期限內做到嗎？ |
| **R** | Relevant 相關 | 這個目標跟你更大的目標相關嗎？做完會推進你到哪？ |
| **T** | Time-bound 有期限 | 截止日是哪一天？什麼時候是中途檢核點？ |

### 拆解方式

- 拆成 5 至 7 個具體任務
- 每個任務有完成標準
- 排上日進度（用艾森豪矩陣排優先順序）
- 估計每個任務的時間（防呆：別超過總工時的 80%）

### PDCA 節奏（每日循環）

| 階段 | 做什麼 | 時點 |
|---|---|---|
| **P** Plan | 完整 SMART 計畫 + 任務清單 | 一開始就設好 |
| **D** Do | 依計畫每日執行 | 每個工作日 |
| **C** Check | 每日結束 5 分鐘檢核 | 每天下班前 |
| **A** Act | 隔天的計畫調整 | 每天早上開工前 |

### 每日 PDCA 檢核腳本（5 分鐘）

每天結束前花 5 分鐘問自己這 4 題：

1. **P 計畫**：今天的計畫是什麼？（看計畫表）
2. **D 執行**：實際完成了哪幾項？（劃掉完成的）
3. **C 檢核**：差距是什麼？為什麼？
4. **A 調整**：明天要補做哪幾項？要刪除什麼？要找誰幫忙？

### 短衝專案的成敗關鍵

- 任務總工時不超過可用工時的 80%（緩衝時間）
- 第 3 天結束時做一次「過半檢核」
- 短衝期內，新任務一律延後
- 容易卡別人的任務先做

## 四、框架 2：OKR × 適應性 PDCA（中長期通用框架）

### 適用情境

1 個月以上的所有中長期目標。例如：
- 三個月完成一本書的初稿
- 半年取得專業認證
- 一年內把品牌從 X 提升到 Y
- 一年內成為某領域被認可的人

### OKR 四件套：完整方法論

OKR 是 Andy Grove 在 Intel 創造、John Doerr 帶到 Google 發揚光大的目標管理方法。本專案包採用 OKR 的「四件套」擴充版：

| 元素 | 後台術語 | **對使用者的說法** | 引導問題 | 數量 |
|---|---|---|---|---|
| 1 | Objective | 「**目標**」或「**你的方向**」 | 你想達成什麼？（質性、振奮） | 1 個 |
| 2 | Key Results | 「**關鍵成果**」 | 怎麼知道達成了？（可量化） | 3-5 個 |
| 3 | Action Items | 「**行動方案**」 | 達成關鍵成果的具體做法？ | 每 KR 配 2-3 個 |
| 4 | Leading Indicators | 「**進度訊號**」 | 怎麼知道有在前進？（過程指標） | 3-5 個 |

**重要：對使用者對話時，使用「對使用者的說法」這欄的中文白話。「OKR」這個縮寫可以用（大眾熟悉），但 Key Results、Action Items、Leading Indicators 等英文不要直接出現**。

### 四個元素的撰寫原則

#### O（目標）撰寫原則

- 振奮人心（讓你早上想跳起來做）
- 質性描述（不要塞數字）
- 有方向感（指向你想成為的樣子）
- 不要太具體（保留執行彈性）

**OK 範例**：「成為閱讀領域被信任的深度評論者」

**不 OK 範例**：「每月寫 4 篇書評」（這是 KR，不是 O）

#### KR（關鍵成果）撰寫原則

- 可量化（一定要有數字或可驗證的結果）
- 有時限（在 X 月底前）
- 具挑戰性（70% 機率達成的難度）
- 跟 O 直接相關

**KR 範例**：
1. Q2 結束前，發表 8 篇深度書評（每篇至少 3,000 字）
2. Q2 結束前，獲得 3 位作家或評論家的公開引用
3. Q2 結束前，建立 1,000 人的訂閱讀者群

#### Action Items（行動方案）撰寫原則

- 具體可動手（不是「努力 X」，是「每週做 Y」）
- 跟 KR 直接關聯（每個 Action Item 對應 1-2 個 KR）
- 數量克制（每個 KR 配 2-3 個 Action Items，避免稀釋執行力）

**Action Items 範例**（對應上述 KR）：
- 為 KR1：每週固定週六上午寫一篇 / 每月找 1 位作家做訪談用作素材
- 為 KR2：每篇文章寄給 3 位該領域作者 / 在 Threads 主動標記與引用
- 為 KR3：每篇文章末附電子報訂閱連結 / 每月做一場 30 分鐘付費讀者 AMA

#### Leading Indicators（進度訊號）撰寫原則

- 是「過程訊號」不是「結果」（會比 KR 早出現）
- 可以「每天/每週」測量
- 跟 Action Items 直接關聯

**Leading Indicators 範例**：
1. 每週寫作小時數
2. 每月文章閱讀量
3. 每週社群互動數
4. 每月推薦轉介數
5. 每篇文章發布後 7 天的訪問數

### OKR 四件套 × PDCA 的整合（核心邏輯）

這是本專案包最重要的設計：**OKR 的四件套，剛好對應 PDCA 的四個階段**。

| PDCA 階段 | OKR 四件套的角色 | 具體做法 |
|---|---|---|
| **P 計畫** | 完整四件套 = 計畫 | 把目標、關鍵成果、行動方案、進度訊號都填好 |
| **D 執行** | 依「行動方案」執行 | 把 Action Items 拆成週任務，每週執行 |
| **C 檢核** | 看「關鍵成果」+「進度訊號」 | 檢核時直接看 KR 達成率 + Leading Indicators 數字 |
| **A 調整** | 調「行動方案」，微調「關鍵成果」 | 不調 O，調 Action Items 或微調 KR |

**為什麼這個整合自然**：

- 一般人「設目標」會卡在「怎麼做」。**Action Items** 補上這個 gap
- 一般人「檢核」會變成「主觀感受」。**Leading Indicators** 讓檢核變成「看數字」
- 一般人「調整」會不知道調什麼。OKR 的結構讓人知道：O 不要調、調的是 **Action Items**

### 拆解方式（三層）

依時間長度不同，拆解粒度不同：

| 時間 | 拆解結構 |
|---|---|
| 1-3 個月 | 雙週 sprint → 週任務 → 本週 3 件事 |
| 3-6 個月 | 月度里程碑 → 週任務 → 本週 3 件事 |
| 6 個月以上 | 季度路線圖 → 月度里程碑 → 週任務 |

### 三種時長的 PDCA 節奏

#### 4.1 OKR × 雙週 PDCA（1 至 3 個月）

**雙週 PDCA 檢核腳本（30 分鐘）**：

1. **P 計畫**：這 2 週的 sprint 目標是什麼？關鍵任務有哪些？
2. **D 執行**：實際完成了幾項？完成率多少？
3. **C 檢核**：
   - 3-5 個關鍵成果的進度？
   - 進度訊號的數字？
   - 跟原本預期的差距？
4. **A 調整**：
   - 下個 sprint 要砍什麼任務？
   - 要加什麼新任務？
   - 哪個行動方案效果不好要換？
   - 整體目標還是這個方向嗎？

#### 4.2 OKR × 月 PDCA（3 至 6 個月）

**月度 PDCA 檢核腳本（60 分鐘）**：

1. **P 計畫**：本月的階段目標是什麼？對應哪幾個關鍵成果？
2. **D 執行**：實際進度？哪些任務達成了，哪些沒達成？
3. **C 檢核**：
   - 每個關鍵成果的累積達成率？
   - 進度訊號的累積數字？
   - 還在軌道上嗎？（不需 100%，70% 算正常）
4. **A 調整**：
   - 下個月的策略要調嗎？
   - 哪些關鍵成果看起來會跑不完？要不要砍？
   - 哪些新機會出現了？要不要加 KR？

#### 4.3 OKR × 月 PDCA + 季末大檢視（6 個月以上）

每月 60 分鐘檢視（同上）+ 季末 2 小時大檢視。

**季末大檢視腳本**：

**1. 整體達成評分**

| 關鍵成果 | 目標 | 實際 | 達成率 |
|---|---|---|---|
| KR1 | | | % |
| KR2 | | | % |
| KR3 | | | % |
| **整體平均** | | | **%** |

- O 達成感受（1 至 10 分）
- 整體完成率判讀：70% 是優秀、太低代表訂太難、太高代表訂太簡單

**2. 這個季度的學習**

- 最大的收穫
- 最大的意外
- 最關鍵的調整

**3. 下季 OKR 規劃**

- O 還是同一個嗎？要不要換？
- KR 要全部新訂，或繼續未完成的？
- 哪些 Action Items 效果好要保留？哪些要換？
- 哪些 Leading Indicators 要新增？

### OKR 的常見誤區（要在實作時避免）

| 誤區 | 為什麼錯 | 本專案包的避免做法 |
|---|---|---|
| 把 KPI 當 OKR | KR 設得保守、100% 達成，失去挑戰性 | 引導使用者「KR 要訂在 70% 機率達成的難度」 |
| O 太具體 | 失去振奮性，變執行清單 | 提供 3 個振奮版本讓使用者指認 |
| KR 太多 | 稀釋執行力 | 限制 3 至 5 個，超過要砍 |
| 沒有行動方案 | 設完 KR 不知道怎麼動 | 強制要求填 Action Items |
| 只看 KR 達成率 | 中途看不到進度 | 強制要求填 Leading Indicators 並追蹤 |
| 從不調整 | 計畫變固定 | PDCA 檢核強制要有 A（調整）的動作 |

## 五、PDCA 循環核心邏輯

PDCA（Plan-Do-Check-Act）是日本豐田汽車、台積電等卓越組織推崇的持續改善方法。原型來自 Walter Shewhart（1939），由 W. Edwards Deming 推廣，後來被製造業、軟體開發、教育、醫療普遍採用。

### 為什麼 PDCA 重要

| 沒有 PDCA | 有 PDCA |
|---|---|
| 計畫做完就放著 | 計畫被定期檢視 |
| 出問題才發現 | 提前識別問題 |
| 重複犯同樣的錯 | 從每次循環學到東西 |
| 計畫跑不完歸咎於「太忙」 | 找出真實原因並調整 |

### 在本專案包的應用

- **每個計畫都有對應的檢核時程**（不可省略）
- **每個檢核點都有可量化的指標**（避免主觀感受）
- **每個檢核都包含調整建議**（不只回顧，更要前進）

## 六、輔助工具

### 6.1 艾森豪矩陣（優先順序）

依「重要 / 緊急」兩軸分四象限：

| | 緊急 | 不緊急 |
|---|---|---|
| **重要** | ① 立刻做 | ② 排程做 |
| **不重要** | ③ 授權或快速處理 | ④ 刪除 |

應用：在任務排序時，請使用者勾每個任務屬於哪一象限。

### 6.2 80/20 法則（帕雷托原則）

20% 的關鍵任務，會帶來 80% 的成果。

應用方式：
- 列出所有任務後，請使用者圈出「如果只能做 20%，會選哪些」
- 這些任務優先安排在精神最好的時段

### 6.3 反向規劃（Backwards Planning）

從終點倒推回今天。

操作步驟：
1. 定義終點狀態（X 月 Y 日達成什麼）
2. 倒推每個里程碑（每個必須完成的中間點）
3. 每個里程碑都是「不做就會失敗」的截止點
4. 從今天到第一個里程碑，倒推本週要動什麼

## 七、可擴充框架（未來新增區）

> 未來如需新增框架，建議放在此區，並在第二節「框架選擇邏輯」加入觸發條件。

**候選擴充清單**：

- **OGSM（Objective, Goals, Strategies, Measures）**：適合需要更強制路徑層的組織決策
- **Hoshin Kanri（方針管理）**：適合超大型多年策略
- **Balanced Scorecard（平衡計分卡）**：適合需要兼顧財務、顧客、流程、學習成長
- **Kanban Board（看板管理）**：適合視覺化的持續性工作
- **GTD（Getting Things Done）**：適合「任務多、領域散」的個人工作流

## 八、給瓦基／學員的客製化指引

如果你要把本檔改成更貼近你的領域，建議優先補充：

1. **加入你領域的目標範例**（在 OKR 範例那一節新增）
2. **調整 PDCA 檢核腳本**（依你的工作節奏調整問題內容）
3. **替換對應的白話用語**（如果你的學員不喜歡「目標 / 關鍵成果 / 行動方案 / 進度訊號」這套說法，可以換成你領域慣用的詞）
