# 常見目標類型與拆解範例 v6

> 本檔提供「策略規劃引擎」在第二、第三階段使用的目標範例庫。AI 會依使用者的角色，從中抽取對應的典型目標與拆解模式。

---

## 設計原則

本範例庫的核心命題：**讓使用者在 30 秒內找到「跟我一樣的目標」，並看到範例拆解後迅速啟動規劃**。

範例呈現要服膺三個原則：

1. **貼近日常**：不要全是「年度大目標」，要涵蓋每週、每月、每季的中小型目標
2. **拆解可複製**：範例的拆解結構，使用者可以直接套用到自己情境
3. **時間區隔清楚**：每個範例都標註對應的時間長度與推薦框架

### 1.2 範例拆解：下週要完成跨部門報告（1 週 / SMART）

**SMART 目標**：
- S：完成一份 15 頁的跨部門協作流程改善報告
- M：包含 5 個具體改善建議、3 個量化的改善成效預測
- A：以目前手上的訪談資料 + 兩個工作日的撰寫時間，可以完成
- R：跟我這季 KPI 的「跨部門效率改善」直接相關
- T：本週五下班前完成初稿，週日完成終稿

**任務拆解**：

| 日期 | 任務 | 完成標準 | 重要/緊急 |
|---|---|---|---|
| 週一 | 整理過去訪談資料 | 5 份訪談記錄整合到一份摘要 | 重要 |
| 週二 | 寫出 5 個改善建議 | 每個 1 段文字 + 1 個圖表 | 重要且緊急 |
| 週三 | 完成改善成效預測 | 3 個量化預測 + 計算依據 | 重要且緊急 |
| 週四 | 撰寫完整報告初稿 | 15 頁完整內容 | 重要且緊急 |
| 週五 | 找同事 review | 至少 2 位給回饋 | 重要 |
| 週六 | 依回饋修改 | 整合所有回饋 | 重要 |
| 週日 | 最終校對 + 排版 | 終稿完成 | 重要 |

**每日 PDCA**：每天下班前 5 分鐘檢核今天完成度，明天的計畫調整

### 1.3 範例拆解：一年內成為部門內某領域的關鍵人物（1 年 / OKR）

**你的目標**：在一年內成為部門內 AI 應用領域的諮詢對象

**關鍵成果**（3 個量化結果）：
1. 年底前主導 3 個 AI 應用導入專案
2. 累積 5 次內部分享（含部門以外）
3. 年底前有 10 位以上跨部門同事主動找你諮詢

**行動方案**（每個關鍵成果配 2-3 個具體做法）：
- 為 KR1：每月找 1 個部門業務流程做 AI 改造試點 / 主動跟主管申請 AI 導入專案
- 為 KR2：每月寫 1 篇內部 AI 工具研究文 / 報名公司內訓擔任講師
- 為 KR3：在公司 Slack 主動回答 AI 問題 / 建立「AI 諮詢」內部頻道

**進度訊號**（4 個過程指標）：
1. 每月研究文章完成數
2. 每月參與專案數
3. 每月主動回答的問題數
4. 季度被同事引用次數

**節奏**：每月 60 分鐘檢視 + 季末 2 小時大檢視

### 2.2 範例拆解：一季新 feature 完整開發（3 個月 / OKR × 雙週 PDCA）

**你的目標**：在 Q2 結束前，把「智慧推薦」功能完整上線並驗證效果

**關鍵成果**：
1. 4 月底前完成原型驗證（PoC）
2. 5 月底前完成全量上線
3. 6 月底前推薦點擊率達到 15%

**行動方案**：
- 為 KR1：先做 5% 用戶 A/B test / 用既有資料訓練演算法雛形
- 為 KR2：分階段擴大（5% → 25% → 100%）/ 同步監控既有功能影響
- 為 KR3：上線後每週調校演算法 / 加 1-2 個個人化參數

**進度訊號**：
1. 每雙週的開發進度（用戶故事完成率）
2. 演算法準確度（每週測一次）
3. 上線後的點擊率 / 跳出率（每日監控）

**任務拆解（雙週為單位）**：

| 雙週 | 階段目標 | 關鍵任務 |
|---|---|---|
| W1-W2 | 演算法設計完成 | 資料分析、模型選型、開發環境設置 |
| W3-W4 | PoC 開發 + 測試 | 5% 用戶測試、修改演算法 |
| W5-W6 | 開發前端介面 + 整合 | UI/UX 完成、後端對接 |
| W7-W8 | 25% 灰度上線 | 監控指標、修 bug |
| W9-W10 | 全量上線 | 全用戶推出、行銷溝通 |
| W11-W12 | 數據驗證 + 優化 | 達成 15% 點擊率、文檔整理 |

**本週 3 件事**：
1. 跟資料工程師確認推薦演算法的訓練資料準備好
2. 跟 UX 設計師確認介面設計稿
3. 跟主管同步整體時程安排

**雙週 PDCA**：每個 sprint 結束花 30 分鐘 review

1. 6 個月內完成 3 個 GitHub 公開的小型實戰專案
2. 在公司內部技術分享會做 2 場相關主題分享
3. 將團隊一個既有專案的關鍵模組用 Rust 重寫並上線

**行動方案**：
- 為 KR1：每週固定週末 4 小時學習實作 / 從官方文件依序做完 3 個範例專案
- 為 KR2：第 3 個月做第一場入門分享 / 第 6 個月做進階實戰分享

**進度訊號**：
1. 每週實作時數
2. 每月 GitHub commit 數
3. 每月參與社群討論次數
4. 內部專案進度

**月度里程碑**：

- **月 1**：完成基礎語法學習 + 第一個小工具
- **月 2**：完成第二個小專案（中型複雜度）
- **月 3**：技術分享會第一場
- **月 4**：選定要重寫的既有模組，完成設計
- **月 5**：重寫實作 + 內部測試
- **月 6**：上線 + 第二場技術分享

**節奏**：每月 60 分鐘檢視

### 4.2 範例拆解：三個月內寫完一本書的初稿（3 個月 / OKR × 雙週 PDCA）

**你的目標**：在 6 月底前完成一本 8 萬字的書稿

**關鍵成果**：
1. 完成 10 章的章節大綱
2. 完成 8 萬字以上的初稿（每章 8,000 字）
3. 有 3 位試讀者讀過並給回饋

**行動方案**：
- 為 KR1：第一週做完整章節腦力激盪 / 第二週收斂為 10 章定稿
- 為 KR2：每週固定 4 個寫作時段（早上 6-9 點）/ 卡關時用「觀點作品引擎」加速
- 為 KR3：每完成一章寄給試讀者 / 每月一次三人線上回饋會

**進度訊號**：
1. 每週寫作時數
2. 每週完成字數
3. 章節推進進度
4. 試讀者回饋次數

**任務拆解（雙週為單位）**：

| 雙週 | 階段目標 | 關鍵任務 |
|---|---|---|
| W1-W2 | 完成章節大綱 | 列出 10 章、每章主旨、核心觀點 |
| W3-W4 | 完成第 1-2 章初稿 | 16,000 字 |
| W5-W6 | 完成第 3-4 章初稿 + 第 1-2 章試讀回饋 | 16,000 字 + 第一波修改 |
| W7-W8 | 完成第 5-6 章初稿 | 16,000 字 |
| W9-W10 | 完成第 7-8 章初稿 | 16,000 字 |
| W11-W12 | 完成第 9-10 章 + 全書回饋整合 | 16,000 字 + 二次修改 |

**雙週 PDCA**：每個 sprint 結束檢視「字數達標率」「卡關章節」

### 4.3 範例拆解：一年內建立穩定的訂閱型收入（1 年 / OKR × 月+季末）

**你的目標**：建立每月 NT$50,000 的穩定訂閱型收入

**行動方案**：

- 為 KR2：每月做 1 次付費讀者 AMA / 內容深度持續提升
- 為 KR3：每月推一次「推薦獎勵」活動 / 每篇內容加推薦連結

**進度訊號**：
1. 每月新訂閱數
2. 每月退訂數
3. 每月推薦數
4. 每篇內容開信率
5. 每月互動率

**季度路線圖**：

| 季度 | 重點 |
|---|---|
| Q1 | 累積免費讀者到 3,000 人，準備付費內容架構 |
| Q2 | 推出付費版，達成 100 位付費訂閱 |
| Q3 | 推出年訂優惠 + 推薦機制，達成 300 位付費訂閱 |
| Q4 | 內容深化 + 留存優化，達成 500 位 |

**節奏**：每月 60 分鐘檢視 + 季末 2 小時大檢視

## 七、目標性質的判斷規則（AI 操作指引）

### 規則 1：先看時間長度

| 訊號 | 對應 |
|---|---|
| 明確說了「兩週內」「一個月內」 | 直接套對應框架 |
| 模糊（「快點」「持續推進」） | 主動詢問選項 |
| 沒提時間 | 主動詢問選項 |

### 規則 2：再看目標性質

| 性質 | 訊號 | 對應框架傾向 |
|---|---|---|
| **執行性**（明確要做出什麼） | 「完成 X 報告」「上線 Y feature」 | 短/中期：SMART 或 OKR |
| **成長性**（要累積能力或品牌） | 「成為 X 領域的人」「建立 Y 影響力」 | 長期：OKR × 月+季 |
| **學習性**（要掌握新東西） | 「學會 X」「能獨立做 Y」 | 依時間：3 個月內 SMART/OKR、3 個月以上 OKR × 月 |
| **探索性**（還不確定能不能做到） | 「嘗試看看 X」「試試 Y 方向」 | **建議先用決策分析引擎再來規劃** |

### 規則 3：依角色關鍵字判斷

| 使用者描述關鍵字 | 優先抽取角色類別 |
|---|---|
| 任務、報告、提案、主管、跨部門 | 上班族（2.1） |
| feature、產品、用戶、上線、sprint、roadmap | PM（2.2） |
| 程式語言、框架、套件、重構、上雲、API | 軟體工程師（2.3） |
| 材料、工法、機構、強度、良率、加工 | 機構工程師（2.3） |
| 試劑、分析、儀器、樣品、實驗、檢測 | 化學分析工程師（2.3） |
| 客戶、漲價、外包、自雇、寫書 | 自雇者（2.4） |
| 接案、合作、頻道、影片、訂閱、粉絲 | 自媒體經營者（2.5） |
| 課程、論文、教授、期刊、研究、實驗室 | 學生／研究者（2.6） |

### 規則 4：拆解粒度的對應

| 時間長度 | 拆解粒度 |
|---|---|
| 1-2 週 | 拆到「日」 |
| 1-3 個月 | 拆到「週」（雙週一個 sprint） |
| 3-6 個月 | 拆到「週」（月度里程碑） |
| 6 個月以上 | 拆到「月」（季度路線圖 + 月度里程碑） |
