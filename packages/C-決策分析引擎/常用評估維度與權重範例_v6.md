## 設計原則

本維度池的核心命題：**在資源、時間、精力、資金、成本與團隊都有限的條件下，協助使用者做出更好的選擇**。

使用者最常遇到的，不是一年一兩次的大型轉折（轉職、搬家、創業），而是：

- **每天**都會做的微決策：這個任務先做還是後做？這封信用 A 語氣還是 B 語氣？
- **每週**都會做的工作決策：要接這個案子嗎？這個功能要納入下週開發嗎？這支影片做哪個主題？
- **每月／每季**都會做的方向決策：產品定位往大眾還是深耕？行銷主力放圖文還是短影音？

本維度池優先服務這些高頻場景。

## 一、常用工作決策維度池（最高頻使用）

這一節列出的維度，適用於使用者**每天或每週都會遇到**的工作決策。

### 1.1 資源消耗類

人的資源有限。每個決策都在分配資源。

| 維度 | 為什麼重要 | 典型權重 |
|---|---|---|
| 時間投入（小時數、週數） | 你願意給這件事多少小時 | 4-5 |
| 精力消耗（腦力 / 體力 / 情緒勞動） | 三種勞動消耗不同的補回時間 | 3-5 |
| 金錢成本 | 直接投入或被消耗的資金 | 3-5 |
| 團隊／人力投入 | 需要拉幾個人進來 | 3-4 |
| 工具與素材成本 | 需要採購什麼或借什麼 | 2-3 |

### 1.2 產出價值類

做完之後得到什麼。

| 維度 | 為什麼重要 | 典型權重 |
|---|---|---|
| 短期 ROI（這月看得到效果嗎） | 立即回報 | 3-4 |
| 長期累積（複利效應） | 半年、一年後還在發酵嗎 | 4-5 |
| 學習價值（做完會變強嗎） | 技能累積 | 3-5 |
| 對個人品牌的加分 | 加分還是稀釋 | 3-5 |
| 對既有資產的活化 | 能不能讓過去做的東西被再用 | 3-4 |

### 1.3 執行風險類

做這件事會碰到什麼。

| 維度 | 為什麼重要 | 典型權重 |
|---|---|---|
| 執行可行性 | 我真的做得到嗎 | 4-5 |
| 與既有任務的相容性 | 會不會打亂現有計畫 | 4-5 |
| 失敗的代價 | 翻車的痛有多大 | 4-5 |
| 可逆性 | 做錯能不能改 | 3-5 |
| 機會成本 | 做這個就不能做什麼 | 3-4 |

### 1.4 個人狀態類

人不是機器。決策還要考慮心情。

| 維度 | 為什麼重要 | 典型權重 |
|---|---|---|
| 心情負擔 | 做這個讓你開心還是煩 | 3-5 |
| 興奮度與熱情 | 你會不會主動推動 | 3-4 |
| 與個人定位／價值觀的契合 | 做完你會更喜歡自己嗎 | 3-5 |

## 二、依職業／角色細分的典型決策情境

針對不同角色，列出最常遇到的日常決策題型。AI 在第一階段判斷使用者角色後，可優先抽取對應類別的維度。

### 2.1 上班族／知識工作者（優先）

最大宗的角色。決策範圍從每日任務到專案推進都有。

| 典型決策 | 優先抽取的維度組合 |
|---|---|
| 接不接這個跨部門專案 | 時間投入 + 學習價值 + 與本職任務相容性 + 對職涯加分 + 心情負擔 + 機會成本 |
| 這週優先 A 任務還是 B 任務 | 短期 ROI + 截止日壓力 + 機會成本 + 對主管期待的吻合 + 心情負擔 |
| 提案用 A 角度還是 B 角度 | 受眾共鳴 + 邏輯說服力 + 與既有共識相容性 + 後續被質疑的承受力 + 製作時間 |
| **分析報告／研究報告要不要納入 X 章節** | 與核心論點相關性 + 篇幅負擔 + 資料佐證強度 + 對讀者價值 + 製作時間 |
| **報告要深入挖 A 觀點還是 B 觀點** | 觀點原創性 + 資料可取得性 + 與既有共識的衝擊度 + 製作時間 + 後續延伸潛力 |
| 報告做精簡版還是詳細版 | 受眾偏好 + 製作時間 + 後續被引用機率 + 主管期待 |
| 加班完成 vs 隔天再做 | 品質 + 隔天精神 + 截止日壓力 + 心情負擔 |
| 用 A 工具還是 B 工具 | 學習曲線 + 長期效率 + 團隊既有工具 + 切換成本 |

### 2.2 Project Manager 產品經理／專案經理

PM 的決策核心是「資源有限下的優先順序與取捨」。

| 典型決策 | 優先抽取的維度組合 |
|---|---|
| 這個功能要做還是不做 | 用戶價值 + 開發成本 + 戰略契合 + 競爭壓力 + 機會成本 |
| **下個衝刺週期（sprint）要先做 A 功能還是 B 功能** | 用戶受益人數 + 開發時間 + 與既有路線圖契合度 + 短期 ROI |
| MVP 上線 vs 繼續打磨 | 用戶反饋價值 + 風險 + 競爭壓力 + 團隊負擔 |
| 自研 vs 外包 vs 採用既有方案 | 成本 + 控制度 + 維護負擔 + 時間到上線 |
| 加人 vs 加時 vs 砍範圍 | 預算 + 截止日 + 品質 + 團隊精神狀態 |
| 重大版本 vs 漸進改版 | 風險 + 用戶體驗連續性 + 開發複雜度 + 行銷話題性 |

> 註：sprint 是軟體開發的「衝刺週期」，通常 1 至 4 週為一個循環。PM 在每個 sprint 開始前要決定這次要做哪些功能。如果使用者用「下個 sprint 排哪個」這種說法，意思就是這個取捨決策。

### 2.3 工程師（軟體／機構／化學分析等專業技術工程師）（新增）

工程師最常做的是「**技術路線取捨**」這類決策。每個選擇都會牽動效能、成本、維護性、與既有系統的相容性。

#### 軟體工程師

| 典型決策 | 優先抽取的維度組合 |
|---|---|
| 用 A 程式語言還是 B 程式語言 | 技術成熟度 + 學習曲線 + 社群支援度 + 與既有系統相容 + 長期維護性 |
| 用 A 框架還是 B 框架（React vs Vue 等） | 學習曲線 + 開發效率 + 效能 + 社群支援 + 與既有系統相容 |
| 用既有套件 vs 自己寫 | 開發時間 + 維護成本 + 客製化空間 + 風險 + 對團隊技術成長 |
| 重構 vs 重寫 vs 漸進改進 | 風險 + 時間 + 對既有功能影響 + 長期可維護性 + 團隊負擔 |
| 上雲 vs 本地部署 | 成本 + 效能 + 維護負擔 + 資料安全 + 擴展性 |
| 同步 vs 非同步架構 | 系統複雜度 + 使用者體驗 + 開發時間 + 維護難度 |
| 微服務 vs 單體架構 | 擴展性 + 維護複雜度 + 部署成本 + 團隊規模適配 |

#### 機構設計工程師

| 典型決策 | 優先抽取的維度組合 |
|---|---|
| 用 A 材料還是 B 材料 | 強度與耐久 + 成本 + 加工難度 + 供應穩定性 + 環保合規 |
| 用 A 工法還是 B 工法 | 良率 + 成本 + 製造時間 + 設備需求 + 與後續組裝相容 |
| 自製 vs 外購 | 成本 + 控制度 + 交期 + 品質一致性 + 後續維護 |
| 設計選 A 結構還是 B 結構 | 強度 + 重量 + 製造難度 + 美觀 + 與其他零件介接 |
| 加大設計餘量 vs 精簡設計 | 安全性 + 成本 + 重量 + 製造良率 |

#### 化學分析工程師／實驗研究人員

| 典型決策 | 優先抽取的維度組合 |
|---|---|
| 用 A 分析方法還是 B 方法（例：HPLC vs GC-MS） | 準確度 + 分析時間 + 設備成本 + 樣品消耗 + 操作複雜度 |
| 用 A 試劑還是 B 試劑 | 靈敏度 + 成本 + 安全性 + 廠商穩定性 + 與既有流程相容 |
| 樣品前處理用 A 還是 B | 處理時間 + 回收率 + 操作步驟複雜度 + 試劑成本 |
| 即時分析 vs 累積批次分析 | 結果及時性 + 試劑利用率 + 設備佔用時間 + 人力負擔 |
| 自建分析方法 vs 採用既有標準方法 | 客製化空間 + 開發時間 + 結果可信度 + 法規驗證可能性 |

### 2.4 自雇者／自由工作者／小型創業者

自雇者的決策核心是「有限資源下的客戶經營與時間分配」。

| 典型決策 | 優先抽取的維度組合 |
|---|---|
| 接 A 客戶 vs 接 B 客戶 vs 兩個都接 | 報酬 + 時間投入 + 客戶關係品質 + 學習價值 + 機會成本 |
| 漲價 vs 維持價格 | 客戶流失風險 + 整體營收 + 自我定位 + 工作品質提升空間 + 可逆性 |
| 自己做 vs 外包 vs 招人 | 成本 + 控制度 + 時間釋放 + 學習價值 + 管理負擔 |
| 主動開發新客戶 vs 深耕既有客戶 | 開發成本 + 既有客戶營收潛力 + 風險分散 + 時間投入 |
| 提升專業 vs 拓展人脈 vs 拓展業務範圍 | 短期 ROI + 長期累積 + 個人能量 + 既有資源活化 |
| **寫書時這個章節要保留還是刪除** | 與主軸相關性 + 篇幅平衡 + 對讀者價值 + 重寫成本 |
| **下一本書要做 A 主題還是 B 主題** | 個人熱情 + 市場需求 + 個人定位契合 + 寫作難度 + 已累積素材 |

### 2.5 自媒體經營者／創作者

| 典型決策 | 優先抽取的維度組合 |
|---|---|
| 要不要接這個品牌合作（單次） | 報酬金額 + 個人品牌契合 + 觀眾接受度 + 製作時間 + 機會成本 + 心情負擔 |
| 下一支主力影片／文章主題 | 觀眾共鳴 + 製作時間 + 個人品牌契合 + 長期累積 + 短期流量 + 學習價值 |
| 出書 vs 出線上課 vs 開新單元 | 製作時間 + 變現潛力 + 個人品牌契合 + 長期累積 + 心情負擔 |
| 圖文 vs 短影音 vs Podcast | 製作時間 + 製作成本 + 平台適配 + 觀眾偏好 + 複用價值 |
| 大眾化 vs 深耕特定族群 | TAM 市場規模 + 競爭飽和度 + 個人品牌契合 + 差異化空間 + 變現潛力 |

### 2.6 學生／研究者／學者

| 典型決策 | 優先抽取的維度組合 |
|---|---|
| 修 A 課 vs 修 B 課 | 學習價值 + 與研究方向契合 + 教授風格 + 課業負擔 + 對未來職涯加分 |
| **研究方向：A 主題深入挖還是 B 主題深入挖** | 學術原創性 + 資料可取得性 + 個人興趣 + 指導教授支持度 + 後續延伸潛力 |
| **論文要不要把 X 章節／實驗納入** | 與核心論述的相關性 + 增加的篇幅負擔 + 增強說服力 + 完成時間 |
| 投 A 期刊 vs B 期刊 | 影響因子 + 接受率 + 審稿時間 + 受眾契合度 |
| 跟 A 教授還是 B 教授 | 研究方向契合 + 教授指導風格 + 經費／資源 + 畢業時間預期 |
| 出國交流 vs 留校精進 | 機會難得性 + 經濟成本 + 進度影響 + 個人成長 + 退場成本 |

## 四、工程決策專屬維度池（新增）

工程決策有它自己的維度語言。當使用者描述偏向技術選型、架構選擇、實驗方法時，優先從本節抽取。

### 4.1 軟體工程類

| 維度 | 為什麼重要 |
|---|---|
| 技術成熟度 | 業界用多久了，坑踩完了沒 |
| 學習曲線陡度 | 團隊接手要花多久 |
| 社群支援度（star、文件、討論度） | 遇到問題求救的容易度 |
| 與既有系統相容性 | 整合的痛感 |
| 長期可維護性 | 5 年後接手者好不好讀 |
| 效能（吞吐量、延遲） | 對使用者體驗的影響 |
| 擴展性 | 流量翻倍能不能撐 |
| 安全性 | 漏洞風險 |
| 部署與維運成本 | 上線後的隱性負擔 |

### 4.2 機構／硬體設計類

| 維度 | 為什麼重要 |
|---|---|
| 強度與耐久性 | 結構安全 |
| 加工難度 | 製造的可行性 |
| 良率 | 量產的成本 |
| 材料／工具成本 | 直接成本 |
| 供應商穩定性 | 缺料的風險 |
| 環保合規（RoHS、REACH 等） | 法規限制 |
| 與後續組裝相容性 | 跟其他零件能不能對接 |
| 售後維修難度 | 客戶服務成本 |

### 4.3 化學分析／實驗類

| 維度 | 為什麼重要 |
|---|---|
| 準確度與精密度 | 結果的可信度 |
| 分析時間 | 一個樣品要跑多久 |
| 樣品消耗量 | 樣品珍貴時很關鍵 |
| 設備成本 | 一次性投資 |
| 試劑成本 | 持續性消耗 |
| 操作複雜度 | 操作員的訓練成本 |
| 法規／標準合規 | 通過驗證的可能性 |
| 廠商支援 | 設備壞了維修的速度 |

## 五、較大型／低頻決策（保留參考）

這類決策每年可能就一兩次，但發生時影響很大。

### 跳槽、轉職、創業
優先納入：收入長期累積 + 職涯發展 + 學習成長 + 心理負擔 + 可逆性 + 退場成本

### 搬家、移民
優先納入：通勤居住影響 + 家庭與伴侶 + 朋友圈 + 生活品質 + 退場成本 + 自我認同

### 結婚、生子、買房
優先納入：可逆性 + 心理負擔 + 長期影響 + 退場成本 + 家庭支持 + 財務影響

## 六、容易被忽略，但要主動納入的維度

無論使用者情境是什麼，這幾個維度建議主動納入評估池：

| 維度 | 為什麼重要 |
|---|---|
| **機會成本** | 選了 A 等於放棄 B 的所有可能。多數人只看 A 的好，不看「放棄 B」的痛 |
| **可逆性** | 不可逆決策必須加倍謹慎 |
| **心情負擔** | 數字算對但心裡撐不住，計畫也跑不完 |
| **與既有任務相容性** | 加一件事看起來合理，但會排擠原本的進度 |
| **隱性成本** | 容易被忽略的副作用（例如熬夜的隔天精神、應酬的人情） |

## 七、維度抽取規則（AI 操作指引）

### 規則 1：先判斷情境類別

| 使用者描述關鍵字 | 優先抽取類別 |
|---|---|
| 任務、報告、提案、加班、跨部門、主管、團隊內部、寫報告、分析報告 | 上班族類別（2.1） |
| feature、功能、產品、用戶、上線、sprint、衝刺週期、roadmap、路線圖 | PM 類別（2.2） |
| **程式語言、框架、套件、重構、上雲、API、架構、後端、前端** | **軟體工程師類別（2.3）** |
| **材料、工法、機構、強度、良率、加工、製造、組裝** | **機構工程師類別（2.3）** |
| **試劑、分析、儀器、樣品、HPLC、GC-MS、實驗、檢測** | **化學分析工程師類別（2.3）** |
| 漲價、外包、招人、開發新客戶、自雇、客戶、寫書、出版 | 自雇者類別（2.4） |
| 接案、合作、報價、頻道、影片、文章、podcast、訂閱、粉絲 | 自媒體經營者類別（2.5） |
| 課程、論文、教授、期刊、研究、出國、研究方向、實驗室 | 學生／研究者類別（2.6） |
| 產品定位、目標市場、TAM、紅海、大眾化 | 產品方向類別（3.1） |
| 內容、影音、圖文、平台、推廣、行銷 | 行銷計畫類別（3.2） |
| 跳槽、轉職、創業、搬家、移民、買房 | 大型決策類別（第五節） |

### 規則 2：必含 1 至 2 個「容易被忽略」的維度（第六節清單）

機會成本、可逆性、心情負擔，至少擇一強制納入。

### 規則 3：不重複

避免維度大幅度重疊。

| ❌ 重複 | ✅ 分開 |
|---|---|
| 「收入」+「短期 ROI」 | 取一即可 |
| 「身心健康」+「心情負擔」 | 取「心情負擔」更具體 |
| 「個人品牌契合度」+「對個人品牌的加分」 | 取一即可 |

### 規則 4：具體不空泛

| ❌ 空泛 | ✅ 具體 |
|---|---|
| 品牌 | 個人品牌契合度 / 個人品牌一致性 |
| 時間 | 時間投入（小時數）/ 製作時間 / 截止日壓力 |
| 風險 | 失敗的代價 / 可逆性 / 機會成本 |
| 技術 | 技術成熟度 / 與既有系統相容性 / 長期可維護性 |

### 規則 5：永遠保留自由補充

在 7 個選項之外，永遠明確邀請使用者增加維度。

### 規則 6：使用者勾選後的處理

- 勾選 3 個：直接進入下階段
- 勾選 4 至 5 個：直接進入下階段
- 勾選超過 5 個：提醒「太多維度會稀釋焦點，請優先排出最重要的 5 個」
- 勾選少於 3 個：提醒「至少 3 個維度才能做出可靠比較，要不要再多挑兩個？」

### 規則 7：Pugh Matrix 情境的微調

如果走 Pugh Matrix（3+ 方案），維度抽取數可以略增到 5 至 7 個。

## 八、典型情境組合範例

這些是日常工作中最常見的決策題型，以及對應的維度組合建議。

### 範例 1：上班族要選 A 提案還是 B 提案（2 案 → 加權矩陣）

2. 邏輯說服力
3. 製作時間
4. 與主管預期的吻合
5. 後續被質疑的承受力
6. 心情負擔（容易被忽略）
7. 學習價值

### 範例 2：上班族的分析報告要不要納入 X 章節（2 案 → 加權矩陣）

抽取的 6 個維度建議：
1. 與核心論點相關性
2. 篇幅負擔
3. 資料佐證強度
4. 對讀者價值
5. 製作時間
6. 機會成本（容易被忽略）

### 範例 3：PM 下個衝刺週期要先做 A 功能還是 B 功能（2 案 → 加權矩陣）

抽取的 7 個維度建議：
1. 用戶受益人數
2. 開發時間
3. 對既有路線圖的影響
4. 競爭壓力
5. 機會成本（容易被忽略）
6. 風險程度
7. 對團隊負擔

### 範例 4：軟體工程師要用 A 框架還是 B 框架（2 案 → 加權矩陣）

### 範例 5：軟體工程師要重構 vs 重寫 vs 漸進改進（3 案 → Pugh Matrix）

## 九、給瓦基／學員的客製化指引

如果你要把本檔改成更貼近你的職業／場域，建議優先補充：

1. **加入你職業專屬的維度**（例如設計師可加「作品集累積」「設計風格契合」；老師可加「學生收穫」「課程準備時間」）
2. **新增你最常遇到的「典型情境組合範例」**（在第八節）
3. **調整典型權重**，讓 AI 在第三階段提出的建議權重更貼近你的價值排序
4. **新增屬於你職業的角色類別**（在第二節新增 2.7、2.8⋯）
5. **替換情境關鍵字**（第七節規則 1 的關鍵字表）讓 AI 能更精準判斷你的情境
